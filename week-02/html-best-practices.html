HTML Best Practices

Develop good habits

There is a preferred method of formatting your code.

It makes life easier for yourself and others who must decipher it later.

Developing good habits when you begin coding can help you avoid problems and bugs, both now and in the long-term when you tackle more complex challenges.

Use a proper DOCTYPE

A DOCTYPE tells a web browser which version of HTML you are using.

We will be using the DOCTYPE indicates our code is written in HTML5.

When you do not use a DOCTYPE, the browser enters what is called quirks mode, which can affect forward/backward compatibility.

This is the DOCTYPE for HTML5:

<!DOCTYPE html>

Declare a character set

A character set tells a web browser how our letters, numbers, special characters should be encoded.

There are many character sets beyond the ASCII standard we discussed when talking about text editors.

Different character sets include many types of glyphs and foreign characters.

We will use UTF-8 since it includes more than 1 million characters.

This is the tag we use to include our character set:

<meta charset="UTF-8" />

Close your tags

Good: <p>Text</p>

Bad: <p>Text

There are a few exceptions which are called self-closing tags. These include:

  • <link />
  • <meta />
  • <img />

Write code in lowercase

HTML is not case sensitive, but...

Writing elements, attributes and values in lowercase is a accepted standard practice in front-end design. It ensures backward compatibility with XHTML (the version of HTML we used before HTML5).

It is also helpful to when trying to discern code from text.

<p class="intro">This is a sentence.</p>

Exception: it is not necessary to use lowercase for alt or title values.

Quote attribute values

Using quotes around attribute values adds visual cues for easier-to-read code.

Quotes also help avoid bugs when working with CSS and Javascript.

<img alt="Sebastian the Ibis" class="photo" src="photo.jpg" />

Attributes should have values

Most attributes require a value. Some do not.

Adding a value, even when not required, will help when working with CSS and Javascript.

<audio controls="controls"></audio>

Comment your code

<!-- This is a comment -->

Adding comments to your code helps you understand why you did what you did.

They also enable whomever inherits your code to understand:

  • the logic behind the program
  • what each part of the program is supposed to do
  • how it is supposed to work

Comments and proper style will aid you in debugging and program maintenance.

Do not use double dashes in comments

<!-- If you must use double dashes in a comment, use two single dashes separated by a space - - like so. -->

Because double dashes are used at the beginning and end of a comment, using them within may invalidate your code or create a bug.

Encode HTML entities

HTML entities are special characters. In code, they begin with an ampersand (&) and end with a semicolon (;).

These may be glyphs or characters used in foreign languages.

Many characters have an entity name and entity number. Most modern browsers will recognize entity names. If the symbol appears broken, however, try using the entity number instead.

Character Code offers an awesome cheat sheet for HTML entities.

The W3C provides a pretty comprehensive reference, too.

Example: HTML Entities

  • Dagger: ‡
  • Spade: ♠
  • Pound Sterling: £
  • Euro: €
  • Copyright: ©

Indent code

HTML is not sensitive to white space.

The use of white space aids readability. It should be used to display the hierarchy of the code.

Indent code by two spaces for each level. This makes it easier to match block-level opening and closing tags, making your code easier to read and maintain.

Most code editors can be set to adjust the indent spacing to utilize the tab key. Make sure it is set to indent with two spaces and not a tab (which is five spaces).

The amount of spacing you use between lines is a matter of preference. Find a pattern which suits you and remain consistent.

Filenames should always be lowercase

The web servers are often case-sensitive.

While your computer may not see the difference between img.jpg and img.JPG, a web server sees these as different files.

<img src="images/miami-beach.jpg" />
